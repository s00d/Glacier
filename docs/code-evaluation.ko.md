# Glacier 코드 평가

작성일: 2026-03-13

## 문서 역할

- 이 문서는 `현재 구현`을 진단하는 문서다.
- 대상은 현재 레포에 있는 코드와 현재 동작 방식이다.
- 목표 제품 요구사항은 [`product-behavior-prd.ko.md`](/Users/fran/Workspace/Glacier/docs/product-behavior-prd.ko.md)에서 다룬다.
- 현재 배포 기준 사용법은 [`README.ko.md`](/Users/fran/Workspace/Glacier/README.ko.md)에서 다룬다.

이 문서는 Glacier를 "코드가 예쁜가"보다 "로직이 적절한가, 더 나은 접근이 있는가, 안정적으로 동작하도록 설계되어 있는가"라는 기준으로 평가한 기록이다.

## 요약

Glacier는 매우 작은 코드로 메뉴바 아이템 숨김 기능을 구현한 흥미로운 AppKit 앱이다. 상태 모델 자체는 단순하고 읽기 쉽다. 하지만 안정성은 상태 머신보다도, 메뉴바 배치와 이벤트 처리를 위해 어떤 macOS 동작에 기대고 있는지에서 크게 좌우된다.

결론부터 말하면:

- 구현은 짧고 명확하다.
- 아이디어는 좋다.
- 공개 API만으로 보장되는 범위 밖의 동작에 많이 의존한다.
- 그래서 "작동하는 프로토타입"으로는 좋지만, "오래 안정적으로 유지될 제품 코드"로 보기에는 위험이 크다.

## 현재 구조

실질적인 런타임 로직은 거의 모두 `Glacier/GlacierController.swift`에 있다.

- `GlacierApp.swift`
  - 앱 진입점
  - `AppDelegate`에서 `GlacierController`를 생성한다.
- `GlacierController.swift`
  - 상태 관리
  - `NSStatusItem` 생성 및 배치
  - 클릭 처리
  - 메뉴 표시
  - 전역/로컬 이벤트 모니터 관리
- `Info.plist`
  - `LSUIElement = true`
  - 독립 창 없이 메뉴바에만 존재하는 앱으로 동작

핵심 메커니즘은 다음과 같다.

1. `●`, `◆`, 그리고 보이지 않는 separator 두 개를 `NSStatusItem`으로 만든다.
2. separator의 길이를 크게 늘려 메뉴바 아이템을 화면 밖으로 밀어낸다.
3. 클릭에 따라 `allHidden`, `partialShow`, `showAll` 상태를 전환한다.
4. 바깥 클릭을 감지하면 다시 숨긴다.

## 잘한 점

### 1. 상태 수가 작고 이해가 쉽다

`allHidden`, `partialShow`, `showAll` 세 상태만 두고 있어 전체 동작을 읽기 쉽다. 이 정도 규모의 앱에서 과한 추상화 없이 바로 따라갈 수 있다는 점은 장점이다.

### 2. AppKit 변경을 메인 액터로 제한한 점은 맞다

`GlacierController`를 `@MainActor`로 선언해 UI 관련 동작을 한 스레드로 묶은 방향은 적절하다. 이 앱은 거의 전부 AppKit 부수효과라서 이 제약이 중요하다.

### 3. 외부 의존성이 없고 구조가 작다

작은 레포에서 동작 원리를 빠르게 파악할 수 있다. 기능이 제한된 대신, 어떤 트릭으로 동작하는지 숨기지 않고 코드에 직접 드러나 있다.

## 핵심 문제

### 1. 공개 API보다 구현 디테일에 많이 기대고 있다

가장 큰 리스크는 `UserDefaults`의 `"NSStatusItem Preferred Position ..."` 키를 직접 조작하는 부분이다.

- 이 키 이름은 코드에 하드코딩되어 있다.
- separator 위치 갱신을 위해 `isVisible`을 껐다 켜며 재계산을 유도한다.
- 이 방식은 문서화된 계약이라기보다 현재 macOS 구현을 이용하는 우회에 가깝다.

즉, 이 코드는 "지금 이 버전의 macOS에서 우연히 잘 맞아떨어지는 방법"일 가능성이 높다. 시스템 업데이트가 들어오면 가장 먼저 깨질 만한 부분이다.

### 2. 상태 전이와 UI 부수효과가 강하게 결합되어 있다

`hideAll()`, `showPartial()`, `showAll()`은 상태 변경 함수이면서 동시에:

- `UserDefaults` 수정
- `NSStatusItem` 위치 변경
- `isVisible` 토글
- separator 길이 변경
- 이벤트 모니터 시작/종료

를 한 번에 수행한다.

지금은 함수 수가 적어서 버티지만, 예외 케이스가 늘어나면 유지보수성이 급격히 떨어진다. 상태 로직만 따로 검증하기도 어렵다.

### 3. 이벤트 정책이 기능 목적과 잘 맞지 않는다

펼친 뒤 마우스 다운을 감시하고 거의 바로 `hideAll()`을 호출하는 구조는 "팝오버 닫기"에는 어울리지만, Glacier의 목적에는 잘 맞지 않는다.

Glacier는 사용자가 펼쳐진 다른 메뉴바 아이템을 실제로 클릭해 쓰게 해야 한다. 그런데 현재 구조는 바깥 클릭을 공격적으로 닫기 이벤트로 간주한다. 이 문제는 UX 이슈이기도 하지만 코드 설계 이슈이기도 하다. 닫기 정책이 앱의 목적과 어긋나 있기 때문이다.

### 4. 윈도우 종류 판별이 취약하다

로컬 모니터에서 `event.window?.className.contains("NSStatusBarWindow")`로 메뉴바 윈도우를 판별한다.

- 문자열 기반 휴리스틱이다.
- 시스템 내부 클래스명에 기대는 셈이다.
- AppKit 내부 구현이 바뀌면 오동작 가능성이 있다.

이런 판별은 "안정적인 분기 조건"으로 보기 어렵다.

### 5. 접근성 처리 로직이 일관되지 않다

코드는 접근성 권한이 없으면 프롬프트를 띄우려 한다. 하지만 앱 설명은 접근성 권한이 필요 없다고 말한다.

이건 두 가지 문제를 만든다.

- 제품 설명과 코드가 모순된다.
- 현재 구현이 실제로 접근성 권한을 필요로 하는지 불명확하다.

권한이 필요 없다면 관련 코드는 제거하는 편이 낫고, 필요하다면 제품 방향부터 다시 정리해야 한다.

### 6. 복구 전략과 운영 안전장치가 거의 없다

이 앱은 메뉴바 위치와 표시 상태를 꽤 공격적으로 건드리는데, 다음과 같은 운영 장치가 없다.

- 레이아웃 초기화
- 실패 시 안전 복귀 경로
- 상태 디버깅 정보
- 사용자 설정 검사나 self-check

한 번 위치가 꼬이거나 사용자가 marker를 헷갈리게 배치하면, 코드 차원에서 회복시켜 줄 수단이 거의 없다.

### 7. 테스트 가능한 구조가 아니다

현재 구조에서는 테스트 대상을 분리하기 어렵다.

- 상태 전이 함수가 AppKit 객체를 직접 만진다.
- 배치 계산이 순수 함수가 아니다.
- 이벤트 정책을 독립적으로 검증할 수 없다.

그래서 동작 확인은 사실상 수동 테스트에 의존하게 된다.

## 더 나은 접근법

Glacier 같은 앱은 목표를 어디에 두느냐에 따라 더 나은 해법이 달라진다.

### 방향 A: 지금 철학 유지

"권한 없이, 작게, 최대한 단순하게"를 유지하고 싶다면 다음이 더 낫다.

#### 1. 자동 닫기 정책을 약하게 만들기

현재처럼 전역 마우스 다운을 즉시 닫기로 쓰지 말고, 아래 중 하나로 제한하는 편이 낫다.

- `●` 재클릭 시 닫기
- `Esc`로 닫기
- 짧은 유예 시간이 지난 뒤 닫기
- 명시적 "모두 숨기기" 액션 제공

기능이 조금 덜 자동화되더라도, 사용 가능한 프로그램이 된다.

#### 2. 상태 계산과 AppKit 적용을 분리하기

예를 들면 다음처럼 나눌 수 있다.

- `GlacierStateMachine`
  - 현재 상태
  - 클릭/옵션 클릭/외부 클릭 입력 처리
  - 다음 상태 계산
- `MenuBarLayoutController`
  - separator 길이 적용
  - 위치 저장/복원
  - status item 가시성 적용
- `EventMonitorController`
  - monitor 시작/종료
  - 어떤 이벤트를 앱 입력으로 해석할지 결정

이렇게 나누면 상태 정책은 테스트 가능해지고, AppKit 해킹 레이어는 최소한 격리된다.

#### 3. 접근성 코드는 제거하거나 명확히 분리하기

현재 수준에서는 접근성 API를 거의 쓰지 않으므로, 프롬프트 코드가 오히려 혼란을 만든다. 이 부분은 없애는 편이 더 안전하다.

### 방향 B: 정말 안정적인 제품으로 가기

"작고 예쁜 코드"보다 "확실하게 동작하는 메뉴바 관리자"가 중요하다면, 접근 방식 자체를 바꾸는 게 맞다.

#### 1. 접근성 계층 사용

메뉴바의 다른 앱 아이템과 상호작용을 정확히 다루려면 다음 같은 정보가 필요하다.

- 실제 어떤 아이템이 보이는가
- 클릭이 어떤 메뉴바 아이템에 들어갔는가
- 어떤 시점에 접어야 하는가

이런 영역은 접근성 계층을 쓰는 편이 더 정석적이다. 구현은 훨씬 무거워지지만 안정성은 올라간다.

#### 2. 레이아웃 모델을 명시적으로 두기

현재는 marker와 separator를 즉석에서 조작한다. 더 안정적으로 가려면:

- visible
- hidden
- alwaysHidden

세 구역을 내부 모델로 먼저 두고, 그 모델을 실제 메뉴바 상태에 투영하는 구조가 낫다.

이렇게 하면 reset, diagnostics, migration 같은 기능도 붙이기 쉬워진다.

## 안정성 관점의 총평

다음 기준으로 요약할 수 있다.

| 항목 | 평가 | 메모 |
|---|---|---|
| 상태 모델 단순성 | 좋음 | 상태 수가 적고 읽기 쉽다 |
| 코드 가독성 | 괜찮음 | 규모가 작아 추적 가능하다 |
| 공개 API 준수 | 낮음 | 시스템 구현 세부사항 의존이 크다 |
| macOS 업데이트 내성 | 낮음 | 위치 저장/재배치 방식이 취약하다 |
| 테스트 용이성 | 낮음 | 상태와 AppKit 부수효과가 결합되어 있다 |
| 장기 유지보수성 | 낮음 | 기능이 늘면 빠르게 취약해질 구조다 |

## 우선순위 제안

지금 코드를 현실적으로 개선하려면 다음 순서가 좋다.

1. 접근성 프롬프트 제거 또는 정당화
2. 자동 닫기 정책 완화
3. 상태 전이와 AppKit 적용 분리
4. 위치/가시성 조작 로직을 별도 레이어로 격리
5. reset 진입점과 진단 로그 추가

## 최종 결론

Glacier의 현재 코드는 "짧고 영리한 AppKit 트릭"으로서는 성공적이다. 하지만 "안정적으로 유지되는 제품 코드"로 평가하면 취약하다.

문제는 코드가 복잡해서가 아니다. 오히려 너무 단순한 대신, 시스템 내부 동작에 기대는 부분이 많고, 기능 정책이 앱 목적과 완전히 맞물려 있지 않다는 점이 핵심이다.

따라서 이 코드는 다음처럼 보는 것이 가장 정확하다.

- 현재 상태: 훌륭한 프로토타입
- 제품 코드로 가기 위한 다음 단계: 이벤트 정책 재설계, AppKit 해킹 레이어 격리, 복구 전략 추가
