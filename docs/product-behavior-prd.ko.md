## 문서 목적

이 문서는 Glacier를 개발자 관점이 아니라 기획자, UI/UX 설계자 관점에서 다시 정의한 제품 요구사항 문서다.

핵심 질문은 다음이다.

- 이 앱은 사용자에게 어떤 가치를 줘야 하는가
- 사용자는 어떤 흐름으로 이 앱을 써야 하는가
- 어떤 상호작용은 허용되어야 하고 어떤 상호작용은 막아야 하는가

이 문서는 구현 방식보다 제품 행동 원칙을 우선한다.

## 제품 정의

Glacier는 메뉴바를 "숨기는 앱"이 아니다. 더 정확히는:

`평소에는 메뉴바를 정리해 두고, 필요한 순간에는 숨겨둔 메뉴바 아이템을 막힘 없이 다시 꺼내 쓰게 해 주는 앱`

이어야 한다.

즉 제품의 핵심 가치는 다음 두 가지를 동시에 만족하는 것이다.

- 평소에는 메뉴바가 정리되어 있어야 한다
- 필요할 때는 숨긴 아이템을 바로 사용할 수 있어야 한다

둘 중 하나라도 무너지면 제품 목적을 실패한 것이다.

## 문제 정의

메뉴바에는 자주 쓰지 않지만 완전히 숨기고 싶지는 않은 아이콘들이 많다.

기존 사용자는 다음 중 하나를 선택해야 한다.

- 메뉴바를 어지럽게 유지한다
- 아이템을 완전히 제거한다
- 복잡한 설정형 앱을 도입한다

Glacier는 이 사이를 메우는 도구여야 한다.

사용자는 이렇게 생각한다.

- "평소엔 안 보여도 된다"
- "필요할 땐 바로 눌러야 한다"
- "배우기 어려우면 쓰지 않게 된다"

## 제품 원칙

### 1. 숨김보다 접근성이 우선이다

숨겨진 아이콘은 사라진 것이 아니라 잠시 뒤로 밀려난 상태여야 한다. 사용자가 다시 꺼냈을 때 바로 클릭하고 사용할 수 있어야 한다.

### 2. 펼쳐진 구역은 작업 공간이다

사용자가 hidden 영역을 펼친 순간, 그 구역은 단순히 보이는 상태가 아니라 실제로 상호작용 가능한 작업 공간이 된다. 이 상태에서는 사용이 끝나기 전까지 앱이 먼저 접히면 안 된다.

### 3. 사용 모드와 편집 모드는 분리한다

일상 사용은 빠르고 단순해야 하고, 구역 재배치는 드물게 수행되는 편집 작업으로 분리되어야 한다.

### 4. 복구 경로는 항상 있어야 한다

메뉴바 레이아웃은 망가질 수 있다. 사용자가 길을 잃지 않도록 reset, 도움말, 편집 재진입 경로가 항상 제공되어야 한다.

### 5. 상태는 눈에 보여야 한다

닫힘, 일부 펼침, 전체 펼침 상태는 사용자가 즉시 구분할 수 있어야 한다.

## 목표 사용자

### 1. 메뉴바 과밀 사용자

- 여러 유틸리티를 항상 켜 두는 사람
- 메뉴바가 길어져 시인성이 떨어진 사람

### 2. 가끔 쓰는 메뉴바 앱이 많은 사용자

- VPN, 클립보드, 캘린더, 볼륨/오디오 도구 등
- 항상 보일 필요는 없지만 빠르게 접근은 해야 하는 사람

### 3. 최소 UI 선호 사용자

- 설정창이 크고 복잡한 앱을 싫어하는 사람
- 메뉴바 도구가 메뉴바를 더 복잡하게 만들지 않길 원하는 사람

## 비목표

현재 버전에서 Glacier는 다음을 목표로 하지 않는다.

- 메뉴바 전체를 완전히 새로 관리하는 대형 프레임워크
- 세밀한 규칙 기반 자동화
- 수십 개 옵션을 가진 설정 앱
- 고급 사용자용 레이아웃 스크립팅 도구

## 핵심 사용자 과업

사용자는 Glacier를 통해 아래 과업을 수행해야 한다.

1. 자주 보지 않아도 되는 메뉴바 아이템을 숨긴다
2. 필요할 때 숨긴 아이템을 빠르게 꺼낸다
3. 펼쳐진 아이템을 실제로 클릭해서 사용한다
4. 사용을 끝내면 다시 깔끔한 메뉴바로 돌아간다
5. 필요 시 구역을 다시 정리한다

## 제품 모드 정의

Glacier는 명확히 두 개의 모드로 동작해야 한다.

### 1. 사용 모드

목적:

- 숨긴 아이콘을 빠르게 꺼내고 사용하기

특징:

- 진입이 빨라야 한다
- 시각적 혼란이 적어야 한다
- 사용 중에는 앱이 방해하지 않아야 한다

주요 액션:

- hidden 영역 펼치기
- always hidden 영역 펼치기
- 펼쳐진 아이콘 클릭/사용
- 접기

### 2. 편집 모드

목적:

- 어떤 아이템이 visible, hidden, always hidden에 속하는지 재배치하기

특징:

- 평소에는 드러나지 않아도 된다
- 대신 진입 후에는 구역이 명확히 보여야 한다
- 실수해도 복구 가능해야 한다

주요 액션:

- marker 위치 조정
- 구역 의미 확인
- 레이아웃 초기화
- 편집 종료

## 핵심 상태 정의

제품 관점에서 사용자가 인지하는 상태는 아래와 같다.

### 1. 기본 상태

- 메뉴바는 정리된 상태
- 컨트롤만 보임
- hidden / always hidden 영역은 접혀 있음

### 2. Hidden 펼침 상태

- 일반적으로 숨겨 둔 아이콘들이 보임
- 사용자는 이 아이콘들을 클릭하고 사용할 수 있음
- 이 상태는 단순 미리보기가 아니라 사용 가능한 상태여야 함

### 3. All 펼침 상태

- always hidden 포함 전체가 드러남
- 자주 쓰지 않지만 유지해야 하는 관리용 상태

### 4. 편집 상태

- marker와 구역 의미가 더 명확하게 드러남
- 사용보다 배치 조정이 우선되는 상태

## 핵심 인터랙션 원칙

### 1. 펼침은 사용 진입이다

사용자가 `●`를 눌러 hidden 영역을 펼쳤다면, 그 다음 행동은 가장 높은 확률로 "펼쳐진 아이콘 중 하나를 클릭하는 것"이다.

따라서 제품은 이 흐름을 기본값으로 가정해야 한다.

허용되어야 하는 흐름:

1. `●` 클릭
2. 숨긴 아이콘 노출
3. 그 아이콘 클릭
4. 메뉴/팝오버/액션 사용
5. 사용 완료 후 정리

막으면 안 되는 흐름:

1. `●` 클릭
2. 숨긴 아이콘 노출
3. 아이콘 클릭 시 Glacier가 먼저 접힘

### 2. 접힘은 사용 종료 뒤에 일어나야 한다

다음은 접힘을 유도해도 된다.

- `●` 재클릭
- `Esc`
- 메뉴바 바깥 빈 영역 클릭
- 일정 시간 무활동 후 자동 정리

다음은 접힘 트리거가 되면 안 된다.

- 펼쳐진 메뉴바 아이콘 클릭
- 펼쳐진 아이콘이 띄운 메뉴/팝오버와 상호작용

### 3. 사용자는 지금 어느 상태인지 즉시 알아야 한다

컨트롤은 최소한 아래를 시각적으로 구분해야 한다.

- 닫힘
- hidden 펼침
- all 펼침
- 편집 모드

표현 방식 예:

- 아이콘 모양 변화
- 채움/윤곽 변화
- 배경 하이라이트
- marker 강조

## 권장 사용자 흐름

### 흐름 A: 첫 실행

목표:

- 5초 안에 앱 개념 이해

요구사항:

- `●`와 `◆`가 무엇인지 알려준다
- visible / hidden / always hidden 구역 개념을 짧게 보여준다
- 편집 진입 방법을 알려준다
- 나중에 다시 볼 수 있어야 한다

권장 방식:

- 첫 실행 1회성 미니 안내
- 우클릭 메뉴의 `사용법`
- `편집 모드` 진입점 제공

### 흐름 B: 일상 사용

목표:

- 숨긴 아이콘을 빠르게 꺼내 쓴다

흐름:

1. 사용자 `●` 클릭
2. hidden 영역 노출
3. 사용자 아이콘 클릭
4. 해당 앱 메뉴/팝오버 사용
5. 종료 시 자연스럽게 다시 정리

성공 조건:

- 사용자가 "꺼내는 과정"을 의식하지 않아야 한다
- 실제 사용 중 Glacier 존재감이 사라져야 한다

### 흐름 C: 드문 편집

목표:

- 어떤 아이콘을 어느 구역에 둘지 재정리

흐름:

1. 우클릭 또는 설정 진입점에서 편집 모드 진입
2. 구역 라벨과 marker 강조 표시
3. 사용자가 항목 재배치
4. 완료
5. 기본 상태 복귀

성공 조건:

- 각 구역 의미가 명확하다
- 잘못 옮겨도 reset이 가능하다

## v1 권장 정책

복잡성을 줄이려면 v1에서는 다음 정책이 적절하다.

- 기본 개념은 `보임 / 숨김` 중심으로 안내
- `always hidden`은 고급 개념으로 노출
- 자동 접힘은 매우 보수적으로 동작
- 편집은 명시적 모드에서만 수행

즉 v1의 핵심은 "적게 숨기더라도 확실히 쓸 수 있게"이다.

## 성공 지표

정량적 수치보다 행동 성공 여부가 먼저다.

핵심 성공 기준:

- 사용자가 펼친 뒤 숨긴 아이콘을 바로 클릭해 사용할 수 있다
- 처음 써 보는 사용자도 구조를 이해할 수 있다
- 메뉴바가 정리되었다고 느낀다
- 레이아웃이 꼬여도 스스로 복구할 수 있다

## 실패 신호

다음 현상이 보이면 제품이 목적을 제대로 이행하지 못하는 것이다.

- 숨긴 아이콘을 누르려는 순간 다시 접힘
- marker 의미를 이해하지 못함
- 편집과 사용의 차이를 모르겠음
- 구역이 꼬였을 때 되돌릴 수 없음
- 앱이 메뉴바를 정리하기보다 더 신경 쓰이게 만듦

## 최종 결론

Glacier는 `메뉴바를 접는 도구`가 아니라 `숨긴 메뉴바를 필요할 때 다시 꺼내 쓰게 하는 도구`로 정의되어야 한다.

따라서 제품 설계의 최우선 원칙은 다음 두 가지다.

1. 펼친 뒤에는 반드시 사용할 수 있어야 한다
2. 사용 모드와 편집 모드는 분리되어야 한다

이 두 가지가 지켜지면 Glacier는 작은 메뉴바 유틸리티로서 분명한 가치를 가진다. 반대로 이 두 가지가 무너지면, 아무리 구현이 영리해도 제품으로서는 실패한다.
